# スタッフ共有機能 設計ドキュメント

作成日：2026-06-04 ／ 対象：Bakery Hub（パン屋向け業務管理SaaS）
ステータス：**設計フェーズ（実装はまだ）**

---

## 1. 何を作るのか（目的）

今は「ログインする人＝お店」という作りで、データ（予約・顧客・商品・在庫）は
すべてその人**個人**にひもづいています。そのため、同じお店をスタッフ同士で
共有できません。

これを「**お店という箱**を作り、その箱に**複数の人**がぶら下がる」形に変えて、
1つのお店を複数スタッフで共有できるようにします。

### 決定済みの方針（石田さんと確認済み）

- **権限は3種類**：オーナー／店長／スタッフ
- **スタッフの加え方**：アプリ内の設定画面で、オーナーがスタッフの
  メールアドレスとパスワードを入力してログインアカウントを作る方式
  （メール送信は使わない）

---

## 2. 用語の整理（この文書での言葉の意味）

| 言葉 | 意味 |
|------|------|
| 店舗（store） | お店そのもの。データの持ち主になる「箱」 |
| メンバー（member） | その店舗に所属するログインユーザー |
| ロール（role） | メンバーの権限。owner / manager / staff の3つ |
| RLS | Row Level Security。DBが「自分の店のデータしか見せない」を強制する仕組み |

---

## 3. 権限（ロール）の設計

| できること | オーナー | 店長 | スタッフ |
|------------|:---:|:---:|:---:|
| 予約・顧客・商品・在庫の閲覧/編集 | ○ | ○ | ○ |
| 販売分析の閲覧 | ○ | ○ | ○ |
| 店舗設定（店舗名・受取時間・定休日） | ○ | ○ | × |
| スタッフの追加・削除・権限変更 | ○ | × | × |
| 店舗の削除 | ○ | × | × |

- **オーナー**：お店の代表。すべて可能。スタッフ管理もできる唯一の役割。
- **店長**：日々の運営＋店舗設定までできるが、スタッフ管理はできない。
- **スタッフ**：日々の業務（予約・在庫・顧客・商品・分析）のみ。

> 根拠：これは一般的なSaaSの権限設計（オーナー＝管理者、中間管理職、一般メンバー）を
> パン屋の運用に合わせて構成したものです。特定の出典に基づくものではなく、
> Claude が一般的な知識をもとに構成した案であり、運用で見直し可能です。

## 4. データベースの変更

### 4-1. 新しいテーブル

```sql
-- お店（テナント）。店舗設定はここに移す
create table stores (
  id uuid primary key default gen_random_uuid(),
  name text not null default '',
  pickup_start time,
  pickup_end time,
  closed_days text not null default '',
  created_at timestamptz not null default now()
);

-- 誰がどの店に、どの権限で所属するか
create table store_members (
  store_id uuid not null references stores(id) on delete cascade,
  user_id  uuid not null references profiles(id) on delete cascade,
  role text not null default 'staff',         -- owner / manager / staff
  created_at timestamptz not null default now(),
  primary key (store_id, user_id)
);
create index idx_store_members_user on store_members(user_id);
```

### 4-2. 既存テーブルの変更

`customers` / `products` / `reservations` / `inventory_logs` の持ち主を
「個人（user_id）」から「お店（store_id）」に変えます。

```sql
alter table customers      add column store_id uuid references stores(id) on delete cascade;
alter table products       add column store_id uuid references stores(id) on delete cascade;
alter table reservations   add column store_id uuid references stores(id) on delete cascade;
alter table inventory_logs add column store_id uuid references stores(id) on delete cascade;
```

- `profiles` は「個人」の情報として残す（表示名など）。
  店舗設定（store_name / pickup_* / closed_days）は `stores` に移動。
- `reservation_items` は親予約（reservations）経由で店を判定するので変更不要。

### 4-3. 既存データの引っ越し（マイグレーション）

今いるユーザー（石田さんのテスト含む）を、それぞれ新しいお店のオーナーにします。

```sql
-- 1) 既存プロフィールごとにお店を作る
insert into stores (id, name, pickup_start, pickup_end, closed_days)
select gen_random_uuid(), store_name, pickup_start, pickup_end, closed_days
from profiles;
-- ※実際は profiles と stores を対応づけるため、下記2)で一時マッピングを使う

-- 2) その人をオーナーとして所属させる ＋ 既存データに store_id を入れる
-- （実装時に profiles.id ↔ stores.id の対応表を作って一括更新する。
--   現状ユーザーが少ないためリスクは小さい）
```

> 注意：マッピングの正確な手順は実装フェーズで確定します。
> 現在の実データはほぼテストデータのみのため、引っ越しの危険性は低いです。

### 4-4. RLS（自分の店のデータだけ見せる）の書き換え

今は `auth.uid() = user_id`。これを「**自分が所属する店のデータ**」に変えます。
RLSが自分自身のテーブルを参照すると無限ループになりやすいため、
**SECURITY DEFINER の補助関数**で所属店リストを返すのが定石です。

```sql
-- ログイン中ユーザーが所属する店のID一覧（RLSを迂回して安全に取得）
create or replace function my_store_ids()
returns setof uuid
language sql
security definer set search_path = public
stable
as $$
  select store_id from store_members where user_id = auth.uid();
$$;

-- 例：products のポリシー
create policy "store products" on products
  for all
  using (store_id in (select my_store_ids()))
  with check (store_id in (select my_store_ids()));
```

- 設定変更やスタッフ管理など**権限が要る操作**は、ロールも併せて確認します
  （例：店舗設定の更新は role が owner か manager のときだけ許可）。

## 5. スタッフのアカウント作成（決定した方式）

オーナーが設定画面で「スタッフのメール＋パスワード＋権限」を入力 → アカウント作成。

### 重要：仕組み上の注意（セキュリティ）

ログインアカウントを**他人の分まで作る**には、Supabase の管理者用キー
（service_role キー）が必要です。このキーは**絶対にブラウザ側に出してはいけません**
（出すと誰でも全データを操作できてしまう）。

そのため次の作りにします。

1. service_role キーは**サーバー側の環境変数だけ**に置く
   （`SUPABASE_SERVICE_ROLE_KEY`。`NEXT_PUBLIC_` を付けない＝ブラウザに出ない）
2. スタッフ作成は**サーバーアクション**で実行し、その中で
   - まず「呼び出した人が本当にオーナーか」を確認
   - OKなら管理者APIで auth ユーザーを作成
   - 作成したユーザーを `store_members` に「自分の店・指定ロール」で追加
3. 新スタッフは、もらったメール＋パスワードでログインできる

### 新規登録トリガーの修正

今は「新規ユーザー＝新しいお店のオーナー」を自動で作っています。
今後は2パターンに分けます。

- **自分で新規登録した人** → 新しいお店を作り、その人をオーナーにする
- **オーナーがスタッフとして作った人** → 新しいお店は作らず、既存店に所属させる
  （登録時のメタ情報で判別。スタッフ作成側で store_members を入れる）

## 8. リスク・確認しておきたいこと

- **service_role キーの取り扱い**：本番でもサーバー側のみに置く運用を徹底する必要あり。
- **RLSの全面書き換え**：全テーブルのポリシーを差し替えるため、入れ替え時に
  一時的にデータが見えなくなる事故が起きやすい。テスト環境で十分確認してから本番へ。
- **既存データの引っ越し**：今はテストデータ中心なので低リスク。
  本番運用後に同じ変更をするのは大変なので、**本番デプロイ前にこの構造へ移す**のが理想。
- **パスワードの初期設定**：オーナーが決めた初期パスワードを、スタッフが
  初回ログイン後に自分で変更できる導線（既存のパスワード変更機能で対応可能）。

## 9. 根拠の明示

- 本ドキュメントの**DB構造・RLS補助関数・ロール設計**は、Supabase / PostgreSQL の
  一般的なマルチテナント設計パターンと、SaaSの権限管理の一般的な考え方をもとに
  Claude が構成した案です。**特定の単一出典に基づくものではありません**。
- service_role キーをサーバー限定にする点は、Supabase 公式の
  セキュリティ上の前提（管理者キーはクライアントに露出させない）に沿っています。
  実装前に `node_modules/next/dist/docs/` および Supabase 公式ドキュメントで
  最新の管理者API（`auth.admin.createUser`）の仕様を確認すること。
